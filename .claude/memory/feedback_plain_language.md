Luna 自述：零编程知识、零计算机基础、零财务知识、零人事管理经验。

**Why**：堆术语她看不懂会挫败感强；翻译成日常话她能直接判断对错。她不是怕复杂，是怕看不懂——只要用大白话讲清楚她完全能拍板复杂决策（这次假期系统重构就是她在主导）。

**How to apply**：
- 进度汇报用日常话：「在数据库里加了 3 张本子和 2 个新栏目」而不是「Phase 0 / Batch A 落库，含 3 张表 + 2 个 ALTER」
- 必须出现的英文/技术词，第一次用时一行话解释（如 RPC=「数据库里写一段帮你算账的小程序」）
- 不要默认她知道：FIFO / RLS / migration / enum / index / additive / idempotent 都要换成大白话
- 选项 / 决策摆给她时，每个选项后面跟"这意味着什么"用她的视角讲：会影响什么、会不会出事、要等多久
- 代码层面（SQL / JSX 内部）不需要照顾这条，写给系统看的不算

**反例（错误示范）**：
> Batch B = 3 个 RPC 函数，spec 里 RPC 用新中文枚举（年假/已通过），但旧数据还是日语。两个写法：(i) RPC 内部用 CASE WHEN ... 兼容两套...

**正例（应该这么说）**：
> 接下来要写 3 段算账小程序：算年假余额、算红日补休、算代休。问题是数据库里有些字段还是日文写法（"承認"/"有休"），新页面要用中文（"已通过"/"年假"）。两个做法：(1)算账小程序同时认两种写法，先这样跑着，等以后再统一；(2)等所有数据改完中文再写。我推荐(1)，因为不用一次改太多东西不容易出错。
