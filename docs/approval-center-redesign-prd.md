# 承认中心（ApprovalCenter）UI 重设计 PRD

## 1. 产品背景

- 系统：早稻田理工塾内部勤怠/工资管理系统（两家关联公司共用：世家学舍、紫陽花教育）
- 技术栈：React 18（函数组件 hooks，**无 TypeScript**）+ Vite + Supabase。**所有样式用内联 style 对象**，没有 Tailwind / CSS modules / styled-components，**请不要引入**
- 图标库：`lucide-react`
- 使用者：管理员（admin）。普通员工看不到这个页面
- 页面定位：admin 一站式处理所有待审批事项；也要能回看最近已处理记录

## 2. 设计主题（theme 对象 `t`）

组件接收 `t` 作为 prop，所有颜色**必须**从 `t` 取，支持深浅两套主题切换：

| key | 含义 |
|---|---|
| `t.ac` | 主强调色（点击/选中/链接） |
| `t.tx` | 主文字 |
| `t.ts` | 次要文字（比 tx 淡一点） |
| `t.tm` | 弱化文字（说明、时间戳） |
| `t.td` | disabled 文字 |
| `t.bgC` | 卡片底色 |
| `t.bgI` | 输入框底色 |
| `t.bd` | 主边框 |
| `t.bl` | 浅边框（分隔线） |
| `t.gn` | 绿（通过/成功） |
| `t.rd` | 红（驳回/危险） |
| `t.wn` | 黄（警告/待处理） |
| `t.we` | 周末底色 |

## 3. 信息架构

顶部一个大标题 "承认中心 + 待审批件数"，下面 **3 个 tab**（各带未读数徽章）：

1. **休假申请**（leave_requests）
2. **换休申请**（day_swap_requests）
3. **交通费变更**（transport_change_requests）

每个 tab 内部结构：
- 待审批区（带类型筛选 chip，如"有給 3｜病假 1｜振替 2"）
- 最近已处理区（最多 30 条）
- 空状态提示

## 4. 数据模型（每条申请必显示的字段）

所有表共用：`id / employee_id / status（申請中/承認/却下）/ created_at / approved_at / approved_by / rejection_reason`

员工信息通过 `employees` 表 join：`name / department`

### 4.1 休假申请（leave_requests）
- `leave_type`：有給 / 病假 / 振替 / 特休 / 产假 等（用 `LEAVE_TYPES` 里的颜色 badge）
- `leave_date`：日期（一条记录 = 一天）
- `is_half_day`：半天 boolean
- `reason`：事由（选填）

**关键**：员工申请连休时会写入**多条记录**（每天一行）。UI 需要把同一员工在 10 分钟窗口内、同类型、同事由的记录**自动分组成"批次卡片"**，合并显示为 "2026-04-20 → 2026-04-24（5天）"，并支持批量通过/驳回或逐条处理。

### 4.2 换休申请（day_swap_requests）
- `swap_type`：休日出勤 / 平日振替
- `compensation_type`：換休 / 代休 / 加班（两个 badge 并排：swap_type + compensation_type）
- `original_date`：原日期
- `swap_date`：换到哪一天（**代休场景可能为空，显示"待定"**）
- `reason`

同样支持批次分组。日期显示格式：`原日期 → 换休日期（或"待定"）`。

### 4.3 交通费变更（transport_change_requests）
- `previous_amount` → `requested_amount`（¥金额）
- `effective_from`：生效日期
- `reason`

**不分组**，每条独立卡片。通过时**需要同时 PATCH `employees` 表**更新 `transport_amount`，按钮文案写"通过并更新"。

## 5. 核心交互

### 5.1 卡片两种形态

**单条卡片**（pending）：
- 左：员工首字头像（圆形，`t.ac` 主色半透明底）
- 中：姓名 + 部门 + 类型 badge + 提交相对时间（"3 分钟前"）/ 日期 / 事由
- 右：驳回按钮（红描边空心） + 通过按钮（绿实心）

**批次卡片**（pending，≥2 条同组）：
- 同上结构 + 一个"X天"计数徽章
- 右侧按钮变成"驳回全部 / 通过全部"
- 可展开：点击 chevron 箭头显示批次内每一条，支持单条驳回/通过
- 日期行显示范围（首日 → 末日）

**已处理卡片**：
- 右侧按钮位置改成状态 pill（✓ 已通过 / ✕ 已驳回）
- 驳回的额外显示红色的"驳回意见"块
- 底部显示"审批人：XX・2026-04-20 14:30"

### 5.2 驳回 Modal

点击驳回按钮弹出居中 modal：
- 标题"驳回申请" + 说明（单条 vs 批量提示不同）
- 多行 textarea 填驳回理由（选填）
- 取消 / 确认驳回（红色实心）按钮

### 5.3 筛选 chip 栏（仅 leave / swap tab）

待审批列表上方：
- `全部 N` / `有給 3` / `病假 1` ...
- 选中项高亮（`t.ac` 主色描边 + 浅底）
- 再次点击同一项取消筛选

## 6. 业务规则 & 边界

- 只有 admin 能访问，不用做权限判断（路由层已控制）
- 批次分组算法（已实现，UI 不用动）：`employee_id | type | reason | 10分钟时间桶` 相同即同组
- 时间格式：相对时间显示"X 分钟前 / X 小时前 / X 天前 / 日期"；绝对时间 `YYYY-MM-DD HH:mm`
- 空状态：用浅绿色底 "暂无待审批的XX申请"
- 响应式：移动端需要优雅折叠（按钮组和标题行可能 wrap）
- 加载态：居中 "加载中..."

## 7. 现有痛点（重设计要解决的问题）

请针对以下痛点提方案：

1. **信息密度 vs 呼吸感**：目前卡片偏扁平拥挤，员工名、部门、类型 badge、时间戳全挤在一行；希望更清晰的视觉层级
2. **待审批 vs 已处理视觉区分弱**：目前只靠边框颜色（黄 vs 灰）区分，不够一眼看穿
3. **批次卡片的"展开/折叠"不够显眼**：chevron 图标小，用户容易错过
4. **筛选 chip 和 tab 切换语义重叠**：tab 已经分了 3 类，chip 又分子类型，视觉上希望更协调
5. **交通费卡片信息特殊**（金额变更），目前用普通卡片模板有点将就，可以专门为它设计
6. **批量操作的危险感不足**：点"通过全部" or "驳回全部"没任何确认，希望视觉上能体现"这是个批量动作"
7. **移动端体验**：右侧按钮组经常换行，希望有更合适的响应式处理

## 8. 必须保留的约束

- React + 内联 style + lucide-react，**不能引入任何 CSS 库**
- 颜色必须从 `t.*` 取，支持明暗主题
- 所有文案中文
- 与系统其他页面保持视觉一致（卡片圆角 10px、主按钮绿/红/主色、字号 11-14 为主）

希望输出：

- 驳回 Modal、空状态、筛选 chip 的细节稿
